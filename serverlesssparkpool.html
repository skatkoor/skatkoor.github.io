Control Tables:

  • A Control Table is a database table that acts as a central reference point within an ETL process or data pipeline.
  • Its primary role is to manage and co-ordinate the execution of data processing tasks.
  • In the context of Data Integration and processing workflows, like the ones involved in Azure Synapse and various data sources, a control table typically holds the metadata about the data being processed rather than the data itself.

Control Table – Purpose and Metadata

  • Job Control: It keeps track of different data processing jobs, their schedules, statuses. This helps in orchestrating the ETL process. (Helps in ensuring they run in the right order).
  • Execution Metadata: It records metadata such as:
    • Job identifiers or names (unique name or identifiers assigned to individual data processing jobs within the ETL flow.)
    • Last Run Timestamps to know when a job was last executed.
    • Status indicators to show if the last run was successful, failed or still in progress.
    • Source filenames or identifiers, especially while dealing with multiple sources or files.
    • Raw counts or data volumes processed to keep a log of how much data was ingested or manipulated.
  • Parameters and Configuration: It might store parameters that might be needed for the execution of specific jobs, like file paths, source system identifiers, or transformation rules.
    • Control Tables do not execute the transformation logic themselves, but rather provide the references or parameters to the actual transformation logic implemented elsewhere.
    • For example, A Control Table contains the data on which Stored Procedure to execute (Which columns to select and which ones to drop or other tasks).
    • When ETL is triggered, the Control Table references the stored procedure responsible for transforming that data.
  • Audit Trail: Enables tracking of what data is moved, transformed, or loaded at any given time.

Interaction between Control Tables and Stored Procedures

  • Before Data Processing: A Synapse Pipeline might consult the Control Table to determine which data sources need to be processed based on the schedule or last execution date.
  • After Data Processing: Once the data is processed, a stored procedure is called to update the Control Table with the new execution timestamp, status of the run, and any other relevant metadata.
  • Error Handling: If an error occurs during processing, a stored procedure might be used to log error details into the control table or a separate error log table.

Actual Data Handling

  • The control table itself does not store actual data from the data sources. The raw data or processed data is usually stored in different tables or data storage systems like Azure Data Lake Gen 2 in our scenario.
  • The control table is strictly for mapping the ETL process, not for storing the data being processed.

Summary

  • In summary, the control table in our ETL workflow is like the conductor of an orchestra, ensuring each section (data source, transformation jobs, etc.) knows when to play its part.
  • The stored procedures are the musicians’ actions, executing the control table’s instructions with precision and contributing to overall performance (The ETL Process).
Server Less Spark Pools When utilizing a Serverless Spark Pool, Azure takes care of managing the underlying infrastructure for us. For instance, when running a notebook on such a pool, Azure guarantees the availability of a specific number of virtual machines in the cloud. As notebook processing begins, Azure seamlessly distributes the workload across multiple virtual machines, ensuring efficient processing. Here's how it generally works: 1. Serverless Nature: Serverless Spark Pools eliminate the need for us to manage the underlying infrastructure. Azure automatically provisions and manages the resources needed to run your Spark jobs. 2. Scaling: Azure dynamically scales the resources allocated to our Spark Pool based on the workload. When we run a notebook or submit a Spark job, Azure automatically allocates the necessary resources to execute the job efficiently. This includes provisioning and managing the virtual machines (VMs) required for processing. 3. Distributed Processing: Spark inherently distributes the processing of your data across multiple nodes (VMs) in a cluster. When you run a Spark job or notebook on a Serverless Spark Pool, Azure takes care of distributing the workload across multiple VMs in the background, allowing for parallel processing and efficient utilization of resources. 4. Auto-scaling: As your workload changes, Azure dynamically adjusts the number of VMs allocated to the Spark Pool to handle the workload effectively. This means that if your workload increases, Azure will allocate more VMs to distribute the processing load, and vice versa. 5. Cost Efficiency: With Serverless Spark Pools, you only pay for the resources you consume while your Spark jobs are running. Azure automatically deallocates resources when they are no longer needed, helping you optimize costs by avoiding idle resources. Creation of a Spark Pool Spark Pool (Cluster): • A Spark pool, also known as a Spark cluster, is a set of computational resources for processing big data workloads. It includes a collection of virtual machines (VMs) or physical machines, depending on the deployment. • This pool represents both the hardware and software infrastructure dedicated to executing Spark jobs and applications. Spark Application: • A Spark application is an instance of a Spark program that gets submitted to a Spark pool for execution. It is the complete program you run using Spark. • The application is the logical sequence of operations consisting of tasks, transformations, and actions performed on distributed data. • Typically, a Spark application is divided into one or more jobs, which are further broken down into tasks that are distributed across the cluster. Java Virtual Machine (JVM): • The Java Virtual Machine (JVM) serves as the platform for running Java applications, converting Java bytecode into machine code for execution on hardware. In Spark, written in Scala (JVM-based), each executor operates within a JVM instance, enabling code execution. • In PySpark, the relationship between JVMs and executors mirrors that of Scala-based Spark applications, albeit with Python's involvement. Each PySpark executor runs within its JVM instance, similar to Scala-based Spark setups. • Tasks are divided and assigned to executors as in Scala-based Spark jobs. Despite PySpark being Python-based, the underlying Spark infrastructure, implemented in Scala, runs on JVMs. PySpark also deploys Python worker processes on executors to execute Python code. • PySpark utilizes the Py4J library to enable seamless communication between Python and JVM-based Spark components. While JVM executes Scala or Java bytecode, Python code is interpreted by Python interpreter in worker processes. • Transformations and Actions: Similar to Scala-based Spark applications, PySpark supports two types of operations: transformations and actions. Transformations are operations that create a new RDD from an existing RDD, such as map, filter, and reduceByKey. Actions are operations that compute a result based on an RDD, such as count, collect, and saveAsTextFile. • In summary, PySpark employs JVM instances for each executor, supplemented by Python worker processes for executing Python code. Py4J facilitates interaction between Python and JVM-based Spark components. Executors and Tasks: • An executor is a process running on a VM in the Spark pool. It is responsible for executing the tasks assigned to it. • Each executor is allocated a certain number of resources (CPU, memory) from the VM. • A single VM can host multiple executors, but each executor operates in its own JVM instance. • For every Virtual Machine in spark environment there is a JVM which is used to convert the code to machine language for the executors. • The JVM translates the Scala/ Java bytecode into machine code that the processor can execute. • Executors can be dynamic or static: o Dynamic Executors: Spark can request more executors from the cluster manager as needed for the workload. If executors are idle, Spark may release them to conserve resources. o Static Executors: A predetermined number of executors are allocated for the Spark job's entire duration, regardless of actual workload demands. RDD stands for Resilient Distributed Dataset, and it's a fundamental data structure in Apache Spark. RDDs are immutable collections of objects that are distributed across multiple nodes in a cluster and can be processed in parallel. Here's a detailed explanation of RDDs: 1. Resilient: RDDs are resilient because they can automatically recover from failures. If a partition of an RDD is lost due to a node failure, Spark can reconstruct that partition by using lineage information (the sequence of transformations that led to the RDD's creation) and recompute the lost partition. 2. Distributed: RDDs are distributed across multiple nodes in a Spark cluster. Each RDD is divided into one or more partitions, and these partitions are distributed across the available nodes in the cluster. This distributed nature allows Spark to process large datasets in parallel, leveraging the computational resources of multiple nodes simultaneously. 3. Dataset: RDDs represent datasets that can be processed in parallel. These datasets can be created from various data sources, including files stored in HDFS, S3, or other distributed file systems, as well as from data stored in memory or external databases. 4. Immutable: RDDs are immutable, meaning that once created, their contents cannot be changed. Instead, transformations applied to an RDD result in the creation of a new RDD. This immutability ensures that RDDs are consistent and can be reliably processed in parallel. 5. Transformations and Actions: RDDs support two types of operations: transformations and actions. Transformations are operations that create a new RDD from an existing RDD, such as map, filter, and reduceByKey. Actions are operations that compute a result based on an RDD, such as count, collect, and saveAsTextFile. 6. Lazy Evaluation: Spark uses lazy evaluation to optimize the execution of RDD operations. This means that transformations on RDDs are not executed immediately when they are defined. Instead, Spark builds up a Directed Acyclic Graph (DAG) representing the sequence of transformations, and the actual computation is performed only when an action is called on the RDD. 7. Cacheable: RDDs can be cached in memory to improve performance. By caching RDDs in memory, Spark can avoid recomputing the same RDD multiple times in a Spark job, leading to faster execution times. • Overall, RDDs provide a powerful abstraction for distributed data processing in Spark, enabling fault-tolerant and parallel processing of large-scale datasets across distributed clusters. They serve as the foundation for building high-level abstractions like Data Frames and Datasets in Spark. A Directed Acyclic Graph (DAG) is a data structure used by Apache Spark to represent the sequence of computations or transformations to be applied to data in a distributed computing environment. Here's a detailed explanation of DAG: 1. Directed Graph: A DAG is a type of graph where edges have a direction associated with them. In the context of Spark, the vertices of the graph represent RDDs (Resilient Distributed Datasets) or stages of computation, and the edges represent dependencies between RDDs or transformations applied to the data. 2. Acyclic: Acyclic means that there are no cycles in the graph, i.e., it is not possible to start at a vertex and follow a sequence of directed edges to return to the same vertex. This ensures that computations in Spark are executed in a well-defined order without circular dependencies. 3. Representation of Dataflow: In Spark, each vertex in the DAG represents a stage of computation, which could be an RDD or a transformation applied to an RDD. Each edge represents a dependency between stages, indicating that the output of one stage is used as input for another stage. 4. Transformation Chain: When transformations are applied to RDDs in Spark, they are not executed immediately. Instead, Spark builds up a DAG representing the sequence of transformations to be applied. Each transformation results in the creation of a new RDD, and edges are added to the DAG to represent the dependency between the RDDs. 5. Optimization and Parallel Execution: Spark uses the DAG representation to optimize the execution of computations. It performs various optimizations, such as pipelining transformations and reordering operations, based on the DAG structure to minimize data shuffling and maximize parallelism. 6. Lazy Evaluation: Spark's lazy evaluation strategy is closely tied to the DAG. Since transformations are not executed immediately when they are defined, Spark builds up the DAG incrementally as transformations are applied. Actual computation is deferred until an action is called on the RDD, at which point Spark uses the DAG to determine the optimal execution plan. 7. Execution Plan: Once the DAG is constructed, Spark uses it to create an execution plan that specifies how to execute the computations efficiently. Spark's execution engine traverses the DAG in a topological order, executing each stage of computation in sequence while respecting dependencies between stages. • Overall, the Directed Acyclic Graph (DAG) is a critical component of Spark's execution model, providing a structured representation of the dataflow and transformations applied to data, which enables efficient and optimized parallel execution of computations in a distributed environment. • Imagine that the grand hall of Spark Pool Pavilion is actually made up of several interconnected rooms, each room representing a virtual space (VM) within the Spark Pool.
