Control Tables

{Control Tables:}
  • A Control Table is a database table that acts as a central reference point within an ETL process or data pipeline.
  • Its primary role is to manage and co-ordinate the execution of data processing tasks.
  • In the context of Data Integration and processing workflows, like the ones involved in Azure Synapse and various data sources, a control table typically holds the metadata about the data being processed rather than the data itself.
{Control Table – Purpose and Metadata}
  1. {Job Control:} It keeps track of different data processing jobs, their schedules, statuses. This helps in orchestrating the ETL process. (Helps in ensuring they run in the right order).
  2. {Execution Metadata:} It records metadata such as:
    • Job identifiers or names (unique name or identifiers assigned to individual data processing jobs within the ETL flow.)
    • Last Run Timestamps to know when a job was last executed.
    • Status indicators to show if the last run was successful, failed or still in progress.
    • Source filenames or identifiers, especially while dealing with multiple sources or files.
    • Raw counts or data volumes processed to keep a log of how much data was ingested or manipulated.
  3. {Parameters and Configuration:} It might store parameters that might be needed for the execution of specific jobs, like file paths, source system identifiers, or transformation rules.
{Audit Trail:}
Enables tracking of what data is moved, transformed, or loaded at any given time.
{Interaction between Control Tables and Stored Procedures.}
  1. {Before Data Processing}: A Synapse Pipeline might consult the Control Table to determine which data sources need to be processed based on the schedule or last execution date.
  2. {After Data Processing}: Once the data is processed, a stored procedure is called to update the Control Table with the new execution timestamp, status of the run, and any other relevant metadata.
  3. {Error Handling}: If an error occurs during processing, a stored procedure might be used to log error details into the control table or a separate error log table.
{Actual Data Handling:}
  1. The control table itself does not store actual data from the data sources. The raw data or processed data is usually stored in different tables or data storage systems like Azure Data Lake Gen 2 in our scenario.
  2. The control table is strictly for mapping the ETL process, not for storing the data being processed.
{Summary:}
  1. In summary, the control table in our ETL workflow is like the conductor of an orchestra, ensuring each section (data source, transformation jobs, etc.) knows when to play its part.
  2. The stored procedures are the musicians’ actions, executing the control table’s instructions with precision and contributing to overall performance (The ETL Process).
